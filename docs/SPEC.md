# System Specification

Current step: Step 2 - Documentation

This document specifies the intended behavior of Async Messaging Service.
It intentionally contains no implementation code.

## Product Positioning

Async Messaging Service is a lightweight asynchronous messaging capability
service for other systems.

It is:

- A message persistence and organization layer.
- Object-storage-first.
- Provider agnostic.
- Asynchronous by design.
- Independent from user, authorization, and business systems.

It is not:

- An instant messaging system.
- A real-time chat server.
- A push notification system.
- A user or permission service.
- A search engine.
- A database replacement.

## Priority Rules

When requirements conflict, use this priority order:

1. Simplicity over feature completeness.
2. Compatibility over maximum performance.
3. Async capability over real-time behavior.
4. Explicit behavior over hidden convenience.
5. AI readability over clever abstraction.

## Core Terms

| Term | Meaning |
| --- | --- |
| Actor | External identity that can send or receive messages. The service stores actor identifiers but does not own actor lifecycle. |
| Message | Immutable content record created once and never modified. |
| Message Reference | Small object that points from an access path to a message object. |
| Mailbox | Actor-owned collection of inbox and sent references. |
| Thread | Message relationship scope used for replies and context. |
| Broadcast | System-originated message targeted to all actors, a tag audience, or explicit actors. |
| State | Independent read-position or thread-state object. |
| Attachment | Object-storage file or large object referenced by a message. |
| Provider | S3-compatible object storage implementation. |

## System Responsibilities

The service must support:

- Actor to actor messages.
- Actor to multiple actor messages.
- System broadcasts.
- Threaded replies.
- Text, file, image, audio, video, and custom payload message types.
- Single and multiple attachments.
- Read state.
- Thread state.
- Message and attachment persistence in object storage.
- Provider abstraction for AWS-compatible, Cloudflare R2-compatible,
  Backblaze B2-compatible, and self-hosted S3-compatible storage.

The service must not implement:

- Login.
- Authorization policy.
- User profiles.
- Actor tag membership.
- Online presence.
- Push delivery.
- Full-text search.
- Real-time socket transport.

## Storage Model

Object storage is the only durable persistence layer. All object keys are part of
the data model and must be generated by one object key builder module.

### Root Prefixes

| Prefix | Purpose |
| --- | --- |
| `messages/` | Immutable message bodies. |
| `attachments/` | Attachment metadata and file object references. |
| `mailboxes/` | Actor inbox and sent references. |
| `broadcast/` | Broadcast message bodies and audience descriptors. |
| `states/` | Actor read state and thread state records. |
| `threads/` | Thread metadata and message references. |
| `operations/` | Idempotency mappings and multi-object write operation records. |

### Identifier Strategy

Use time-sortable identifiers for messages, attachments, references, threads,
broadcasts, and state records.

Preferred identifier:

- UUIDv7, because RFC 9562 standardizes a timestamp-first UUID format.

Fallback:

- ULID or another provider-neutral, time-sortable unique identifier when the
  chosen runtime lacks reliable UUIDv7 support.

Identifier requirements:

- No central sequence service.
- Lexicographic or timestamp-derived ordering.
- Safe for object keys after normalization.
- Enough entropy for distributed writers.

### Key Naming Rules

All key segments must be explicit and stable.

Recommended conventions:

- Use lowercase root prefixes.
- Use UTC date buckets such as
  `year=2026/month=06/day=01/hour=11/minute=22` for cursor-friendly access
  paths.
- Normalize actor, tag, and external identifiers before using them in keys.
- Do not allow raw `/`, control characters, or provider-sensitive path segments
  inside external identifiers.
- Do not create empty "folder marker" objects unless a provider requires them.
- Avoid global list operations.

### Planned Object Key Shapes

Message body:

```text
messages/year=YYYY/month=MM/day=DD/hour=HH/minute=mm/{messageId}.json
```

Message direct lookup:

```text
messages/by-id/{messageId}.json
```

Attachment metadata:

```text
attachments/metadata/year=YYYY/month=MM/day=DD/hour=HH/minute=mm/{attachmentId}.json
```

Attachment direct lookup:

```text
attachments/by-id/{attachmentId}.json
```

Attachment object:

```text
attachments/objects/year=YYYY/month=MM/day=DD/hour=HH/minute=mm/{attachmentId}/{normalizedFileName}
```

Actor inbox reference:

```text
mailboxes/{actorStorageId}/inbox/year=YYYY/month=MM/day=DD/hour=HH/minute=mm/{feedSortKey}_{createdAtUtc}_{messageId}_{referenceId}.json
```

Actor sent reference:

```text
mailboxes/{actorStorageId}/sent/year=YYYY/month=MM/day=DD/hour=HH/minute=mm/{feedSortKey}_{createdAtUtc}_{messageId}_{referenceId}.json
```

Thread metadata:

```text
threads/{threadId}/metadata.json
```

Thread message reference:

```text
threads/{threadId}/messages/year=YYYY/month=MM/day=DD/hour=HH/minute=mm/{threadSortKey}_{createdAtUtc}_{messageId}_{referenceId}.json
```

Broadcast body:

```text
broadcast/messages/year=YYYY/month=MM/day=DD/hour=HH/minute=mm/{broadcastId}.json
```

Broadcast direct lookup:

```text
broadcast/by-id/{broadcastId}.json
```

Broadcast audience descriptor:

```text
broadcast/audiences/{audienceType}/{audienceKey}/year=YYYY/month=MM/day=DD/hour=HH/minute=mm/{feedSortKey}_{broadcastId}.json
```

Actor read state:

```text
states/{actorStorageId}/events/messages/{messageId}/year=YYYY/month=MM/day=DD/hour=HH/minute=mm/{feedSortKey}_{stateId}.json
```

Actor thread read position:

```text
states/{actorStorageId}/events/threads/{threadId}/year=YYYY/month=MM/day=DD/hour=HH/minute=mm/{feedSortKey}_{stateId}.json
```

Current state projection:

```text
states/{actorStorageId}/current/{messages|threads}/{targetId}.json
```

Operation records:

```text
operations/idempotency/{callerStorageId}/{idempotencyKey}.json
operations/by-id/{operationId}/started.json
operations/by-id/{operationId}/steps/{stepId}.json
operations/by-id/{operationId}/completed.json
```

The exact key format may be refined during implementation, but normal access
paths must remain prefix-based and must not require full-bucket scans.

## Domain Object Requirements

### Message Object

Required fields:

- Message identifier.
- Sender actor identifier.
- Recipient actor identifiers or audience descriptor.
- Message type.
- Payload reference or inline payload.
- Attachment identifiers.
- Thread identifier when applicable.
- Parent message identifier when applicable.
- Created timestamp.
- Storage schema version.

Rules:

- Message objects are immutable.
- Message objects do not store read state.
- Message objects do not store authorization decisions.
- Message objects may reference attachments but must not embed large file bytes.

### Message Reference Object

Required fields:

- Reference identifier.
- Message identifier.
- Message object key.
- Actor identifier or thread identifier for the owning access path.
- Direction or reference kind.
- Created timestamp.

Rules:

- Reference objects may be duplicated across access paths.
- Reference objects are small.
- Reference objects are the primary query surface.

### Thread Object

Required fields:

- Thread identifier.
- Root message identifier when available.
- Created timestamp.
- Optional external correlation identifier.

Rules:

- Thread membership is represented by thread message references.
- Replies may be multi-level through parent message identifiers.
- Thread state is stored separately.

### Broadcast Object

Required fields:

- Broadcast identifier.
- Audience type: `all`, `tag`, or `explicit`.
- Audience descriptor.
- Message payload.
- Created timestamp.

Rules:

- Broadcast message bodies are not copied per recipient.
- Explicit audience broadcasts may create mailbox references.
- Tag audience broadcasts do not imply this service manages tag membership.

### State Object

Required fields:

- State identifier.
- Actor identifier.
- State kind.
- Target message or thread identifier.
- Read position or read timestamp.
- Created timestamp.

Rules:

- State objects are independent from message objects.
- V1 should use append-only state records.
- A future materialized current-state object is allowed only if the write
  behavior is explicit and guarded by provider-supported preconditions.

### Attachment Metadata Object

Required fields:

- Attachment identifier.
- Object key.
- Original file name.
- Normalized file name.
- Content type.
- Size.
- Checksum when available.
- Created timestamp.

Rules:

- Large files must use provider-native multipart upload when available.
- ETag must not be treated as a universal MD5 checksum because multipart uploads
  and provider-compatible systems may use different ETag semantics.
- Message objects store attachment references only.

## Conceptual API Surface

The implementation API shape is not chosen yet. The service should expose these
capabilities through the selected transport.

| Capability | Input | Output |
| --- | --- | --- |
| Send Actor Message | Sender actor, recipients, message type, payload, attachments, optional thread. | Message identifier, references, object keys. |
| Send Broadcast | System sender, audience descriptor, message type, payload, attachments. | Broadcast identifier, object keys, optional reference count. |
| List Mailbox | Actor identifier, direction, date or cursor, page size. | Ordered message references and next cursor. |
| Get Message | Message identifier or message object key. | Message object and attachment references. |
| List Thread | Thread identifier, cursor, page size. | Thread message references and message summaries. |
| Mark Message Read | Actor identifier, message identifier, timestamp. | State identifier. |
| Mark Thread Read | Actor identifier, thread identifier, read position. | State identifier. |
| Prepare Attachment | Actor or caller context, file metadata. | Upload target or object key. |
| Complete Attachment | Attachment identifier, object key, checksum, size. | Attachment metadata reference. |

## Provider Capability Requirements

The storage port must express provider-neutral operations:

- Put object.
- Put object with create-if-absent precondition when supported.
- Get object.
- Head object.
- List objects by prefix with pagination.
- Delete object when explicitly needed.
- Multipart upload or delegated SDK multipart upload for large attachments.

Each adapter must document:

- Endpoint format.
- Region handling.
- Authentication behavior.
- Conditional write support.
- Multipart behavior.
- List behavior.
- Checksum behavior.
- Known incompatibilities.

## Consistency Expectations

The core service should be designed for providers that offer strong read-after-
write and strong list consistency, but adapters must not hide provider-specific
differences.

Rules:

- Immutable object writes should use create-if-absent semantics when supported.
- The service must tolerate retry after transient write or list failures.
- Caller-visible operations should return object keys so reads do not rely on
  broad discovery.
- Cache layers must be treated as separate from object storage consistency.

## Error Categories

Use explicit error categories:

- Validation error.
- Unsupported provider capability.
- Object already exists.
- Object not found.
- Storage unavailable.
- Storage timeout.
- Partial write failure.
- Serialization failure.
- Configuration error.

## Versioning

All stored object payloads must include a schema version. Future migrations
should prefer additive fields and new object versions over destructive rewrites.

## Non-Goals For V1

- Strong cross-object transactions.
- Global unread-count accuracy.
- Full-bucket search.
- Real-time presence.
- Push notification dispatch.
- Database-backed query projections.
- Provider-specific optimization leaking into domain modules.
